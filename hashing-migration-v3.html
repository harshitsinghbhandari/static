Hashes in AO

Where the codebase hashes, where it counts, and why both are right.

AO uses SHA-256 in five small places — each chosen for collision avoidance against an inherently unbounded input domain. Everywhere else, identifiers are human-readable counters, slugs, or basenames, kept disambiguous by reservation, atomic file creation, or remote ground truth.

5 SHA-256 sites 1 MD5 site (compat) 1 HMAC-SHA256 (signing) Hash lengths: 10 / 12 / 16 / 64

Reading guide

Each hash answers a different question. Some make a bounded namespace from an unbounded one (project storage). Some dedupe (feedback reports). Some fence off cache key collisions (gh wrapper). One mirrors a third-party tool (kimi md5). The non-hash sites are louder than the hash sites — that's the interesting part.

5identity hashes
10–16truncation chars
0random session IDs
1md5 compatibility shim

1. The Philosophy

AO's identifier strategy follows one rule: hash only when the input domain is unbounded and there is no human-friendly disambiguator that survives. Everywhere a human will read or type the identifier — session IDs, branch names, tmux names — keep it readable and resolve collisions by reservation rather than entropy.

Hash when

The input is a path or URL

  • Filesystem paths can be arbitrarily long, contain spaces, slashes, unicode.
  • Origin URLs vary in scheme/host/punctuation for the same logical repo.
  • A 10–12 char SHA-256 prefix is short enough to embed in directory names and long enough to make practical collision irrelevant for one-user / one-machine scale.
  • Determinism matters: the same project must always resolve to the same on-disk slot across re-clones, re-registrations, and machine reboots.
Don't hash when

A human reads or types it

  • Session IDs (ao-79) appear in the dashboard, tmux, branch names, and PR titles.
  • Branch names land on GitHub and need to be readable in PR lists.
  • tmux session names are typed at the terminal during attach/kill.
  • Disambiguation here is solved by reservation: an atomic file lock on sessions/{id}.json + a remote ls-remote sweep.

3. The Non-Hash Map — Identifiers That Stay Readable

These places construct identifiers that humans will read. Each has a different disambiguation strategy. None of them use entropy.

Session ID

{sessionPrefix}-{N} · monotonic counter

e.g. ao-79, int-3. Allocated atomically in reserveNextSessionIdentity.

  • Disambiguation: reserveSessionId(sessionsDir, id) is an atomic create of sessions/{id}.json. O_EXCL semantics — first writer wins.
  • Remote awareness: listRemoteSessionNumbers at session-manager.ts:771-790 sweeps git ls-remote --heads origin "session/{prefix}-*" so a re-cloned project doesn't reuse session numbers that already shipped as branches.
  • Why no hash: users type these into tmux, see them on PR titles, grep for them in logs.

core/src/session-manager.ts:803-832

Session prefix

Heuristic from project ID, not hashed

  • ≤4 chars: the projectId itself, lowercased
  • CamelCase: uppercase letters joined (PyTorch → pt)
  • kebab/snake: initials of each segment (agent-orchestrator → ao)
  • single word: first 3 chars (integrator → int)

Optimized for readability per project, not global uniqueness. Two projects with the same prefix coexist because session IDs are scoped per-project storage directory.

core/src/paths.ts:64-82

tmux session name

Identical to the AO session ID

  • generateSessionName(prefix, num) = `${prefix}-${num}` — same as the session ID.
  • Legacy generateTmuxName prepended {storageKey}- for cross-project disambiguation but is now @deprecated — session prefixes are treated as globally unique.
  • Used directly as tmux new-session -s {sessionName}.

core/src/paths.ts:210-212 · plugins/runtime-tmux/src/index.ts:58-67

Worktree directory

{projectWorktreesDir}/{sessionId}

  • Just the session ID under the project's worktree root. No hash.
  • Disambiguation comes for free: session IDs are already unique within the project.
  • clearStaleWorktreePath at workspace-worktree/src/index.ts:84-99 only deletes paths git no longer registers — preserves user state.

plugins/workspace-worktree/src/index.ts:162

Branch name

Three-layer fallback, all human-readable

  • 1. Explicit spawnConfig.branch wins.
  • 2. From issue: tracker plugin's branchName(issueId, project) (e.g. tracker-github sanitizes the issue title) — only if isGitBranchNameSafe.
  • 3. From free-text issueId: lowercased [a-z0-9-] slug, max 60 chars, prefixed feat/.
  • 4. Final fallback: session/{sessionId}.

No hash. Branch names land on GitHub and must remain greppable. Collision is impossible because {sessionId} already is.

core/src/session-manager.ts:1180-1201

Orchestrator session

{sessionPrefix}-orchestrator + numeric suffix on collision

  • Singleton per project. Branch is orchestrator/{sessionId}.
  • Detection regex: ^{prefix}-orchestrator-\d+$ for re-spawned orchestrators.

5. Patterns and Inconsistencies

Three patterns recur. One inconsistency stands out.

1 Truncation length tracks blast radius 10 chars (project ID, intra-user keyspace), 12 chars (storage key, lifecycle dedupe), 16 chars (feedback dedupe + gh cache). Wider scope ⇒ wider hash.
2 Inputs are normalized before hashing Origin URLs go through normalizeOriginUrl (storage-key.ts:10-31). Evidence strings strip volatile fields. Feedback inputs are lowercased, evidence sorted. Every hash site is deterministic across noise.
3 SHA-256 is the default; MD5 only for compat The single MD5 site (kimicode) exists because kimi itself uses MD5. Comments warn loudly. Nothing AO owns is keyed by MD5.
4 Inconsistency: two SHA-256 hashes for "the project" generateConfigHash (paths.ts:32) hashes the local config directory; generateExternalId (global-config.ts:47) hashes path + origin. Both are 10–12 chars of SHA-256 and both identify "this project," but they aren't interchangeable. The legacy one survives behind @deprecated markers until ao migrate-storage finishes the cutover.

6. RCA — Why a Single User Has Both ao and ao2

A real-world artifact pulled from a live ~/.agent-orchestrator/config.yaml: two checkouts of the same repo registered weeks apart, prefixes ao and ao2. The current canonical collision logic produces ao-1, not ao2. The prefix exists because of three competing collision-suffix schemes that disagree with each other.

Symptom

Two prefixes that shouldn't both be possible

  agent-orchestrator:
    projectId: agent-orchestrator
    path: /Users/.../side-quests/agent-orchestrator
    registeredAt: 1776846929   # 2026-04-22 14:05
    storageKey: 7dc54da05c9e   # ← V1 era
    sessionPrefix: ao

  agent-orchestrator_168566536d:
    projectId: agent-orchestrator_168566536d
    path: /Users/.../main-quests/agent-orchestrator
    registeredAt: 1778140403   # 2026-05-07 13:23 (today)
    displayName: agent-orchestrator-2
    sessionPrefix: ao2          # ← shouldn't exist under current canonical code

Same repo, two clones. The canonical allocator (deriveAvailableSessionPrefix at global-config.ts:619-624) would have produced ao-1. Something else fired.

Smoking gun

A second, no-dash collision counter in start.ts

Inside addProjectToConfig at cli/src/commands/start.ts:688-697:

let prefix = generateSessionPrefix(projectId);
const existingPrefixes = new Set(
  Object.values(config.projects).map(
    (p) => p.sessionPrefix || generateSessionPrefix(basename(p.path)),
  ),
);
if (existingPrefixes.has(prefix)) {
  let i = 2;
  while (existingPrefixes.has(`${prefix}${i}`)) i++;
  prefix = `${prefix}${i}`;     // ← no dash, counter starts at 2
}

Pure string concat: ${prefix}${i}. No separator, counter starts at 2, completely independent from deriveAvailableSessionPrefix in global-config.ts.

Walkthrough

What actually ran when the second clone was registered

1 projectId collision (dashed) start.ts:642-650. basename(path) = agent-orchestrator, already in config. Walks ${id}-${i} from i=2 → projectId = "agent-orchestrator-2".
2 Prefix derivation start.ts:688. generateSessionPrefix("agent-orchestrator-2") sees a kebab-case input, takes initials of agent/orchestrator/2 → ao2. The collision-suffix's "2" leaks into the prefix as a literal character.
3 Prefix collision check skipped start.ts:696-700. ao2 isn't in existingPrefixes, so the no-dash counter loop doesn't fire. Prefix stays ao2.
4 Hand-off with explicit prefix start.ts:721-727. Calls registerProjectInGlobalConfig(..., { sessionPrefix: "ao2" }). The pre-computed prefix is passed in via localConfig.
5 Canonical allocator never runs global-config.ts:708-711. The cascade is existing?.sessionPrefix ?? localConfig?.sessionPrefix ?? generateSessionPrefix(basename(path)). localConfig.sessionPrefix = "ao2" is non-null, so the basename(path) fallback (which would have produced ao → collision → deriveAvailableSessionPrefix → ao-1) never runs.
6 V2 ID swap, prefix preserved global-config.ts:668-685. The projectId gets replaced with generateExternalId(path, origin) = agent-orchestrator_168566536d. name="agent-orchestrator-2" survives as displayName; sessionPrefix="ao2" survives verbatim.

Three (really four) collision-suffix conventions in the same repo

Site Format Start Lives in
addProjectToConfig — projectId ${id}-${i} 2 cli/src/commands/start.ts:642-650
addProjectToConfig — prefix ${prefix}${i} (no dash) 2 cli/src/commands/start.ts:688-697
deriveAvailableSessionPrefix ${prefix}-${suffix} 1 core/src/global-config.ts:610-628
Orchestrator re-add (start.ts) ${id}-${randomBase36(4)} n/a cli/src/commands/start.ts:1460-1470

Three of these allocate human-typed identifiers. Each one disagrees with the others on separator and starting index. The fourth uses entropy.

Deeper bug

deriveAvailableSessionPrefix is dead for the CLI path

Two of the three real entry points pre-compute their own prefix and pass it as localConfig.sessionPrefix to registerProjectInGlobalConfig:

  • ao start <path> via addProjectToConfig (start.ts:725) — uses no-dash counter.
  • registerFlatConfig on first start (start.ts:140-148) — also pre-computes.
  • Web POST /api/projects (web/src/app/api/projects/route.ts:103) — does not pre-compute, only sends defaultBranch.

The cascade in registerProjectInGlobalConfig means localConfig.sessionPrefix wins over the basename(path) fallback. deriveAvailableSessionPrefix only runs through that fallback. So in practice, only the web modal exercises it. CLI registrations bypass it entirely. You can register the same repo through CLI and web and get different prefixes for it.

Fix shape

Centralize allocation; delete the duplicates

  • Delete start.ts:688-697 (the no-dash counter). Stop passing sessionPrefix from addProjectToConfig into registerProjectInGlobalConfig.
  • Same for registerFlatConfig at start.ts:140-148 — drop the pre-computed prefix.
  • Let deriveAvailableSessionPrefix in global-config.ts be the only allocator. Pick a format (${prefix}-${i} reads cleaner) and apply it everywhere.
  • The orchestrator random-suffix at start.ts:1460-1470 is a different concern (cross-orchestrator races) and can stay, but should also funnel through one helper.
  • Existing ao2 entries don't need migration — they're valid prefixes, just inconsistent. New registrations would converge.

Smoking artefact #3

The 153 observability dirs are a hash leak

getObservabilityBaseDir(configPath) at paths.ts:160-163 returns ~/.agent-orchestrator/{generateConfigHash(configPath)}-observability. The hash input is dirname(realpath(configPath)). Every PR session worktree ends up with its own agent-orchestrator.yaml at a unique path, so each session that writes observability data produces a brand-new directory that nobody ever GCs.

The user has 8 projects but 153 observability directories. That ratio is not going to improve — every new PR session adds another one. The data inside is essentially process-level traces (session-manager-{pid}.{json,ndjson}), scoped by PID, not by anything semantic.

9. The Full Problem List

Consolidating everything from the on-disk forensics, the RCA, the dead-code census, and the prior section's hash map. Numbered so the migration plan can reference items.

Identity sprawl

P1 — Three project-identity formats

  • Bare basename (V1, e.g. agent-orchestrator).
  • V2 hashed {basename}_{sha10}.
  • deriveStorageKey origin-anchored 12-char (dead but exported).

Same user has 1 of each kind for the same repo.

Allocation chaos

P2 — Four prefix-collision schemes

  • ${prefix}-${i} from 1 (global-config.ts:620).
  • ${prefix}${i} from 2 (start.ts:697) — produced ao2.
  • ${id}-${i} from 2 (start.ts:644).
  • Random base36 (start.ts:1465).

CLI vs web give different prefixes for same input.

Tmux name drift

P3 — Three tmux conventions live in real metadata

  • {prefix}-{N} — current.
  • {prefix}-{prefix}-{N} — doubled, in ao-ao-orchestrator.
  • {storageKey}-{name}-{prefix}-{N} — legacy, still in orchestrator.json.

Metadata drift = stuck orchestrators.

Hash leak

P4 — 153 observability dirs and counting

One per unique configPath hash. Every PR session creates a new one. No GC.

Stranded data

P5 — Legacy worktree tree, .migrated stragglers

~/.worktrees/agent-orchestrator/ao-101 never adopted. 111111111114.migrated sibling never collected.

Cross-cutting bug

P6 — localConfig.sessionPrefix short-circuits canonical allocator

CLI paths pre-compute prefix and pass it via localConfig; the cascade in registerProjectInGlobalConfig respects it before the canonical allocator can run.

Schema drift

P7 — storageKey still in Zod schema

Optional field, written by no one, present in 2 of 8 user entries. Reads as live state to anything that walks the schema.

Dead exports

P8 — 12 zero-caller helpers in @aoagents/ao-core

Listed in §8. They type-check, they ship, they confuse readers.

Race-prone allocation

P9 — Session counter scans every time

reserveNextSessionIdentity scans sessions/ + git ls-remote on every spawn. O(n) per spawn, no monotonic guarantee if a branch was deleted upstream.

Collision throw

P10 — generateExternalId 10-char hash with no recovery

Throws "extremely unlikely please file a bug" on collision. 40 bits = ~1M before birthday risk; fine per-user, fragile in shared/multi-tenant.

Case sensitivity

P11 — Inconsistent case normalization

Origin URLs lowercased; basenames lowercased in sanitizeBasename; raw paths not lowercased before hashing in generateConfigHash. APFS users can produce two different hashes that map to one directory.

Cross-machine

P12 — File lock is local-only

Synced ~/.agent-orchestrator/ (Dropbox/iCloud) breaks: two machines registering concurrently both think they got the lock.

1 — Session prefix sessionPrefix Allocated once at registration. Default = generateSessionPrefix(displayName); collision against other registered prefixes resolved by one canonical allocator: ${prefix}-${i} starting at i=1. Stored in identity.json. identity.json.
2 — Session ID sessionId {sessionPrefix}-{N} where N is monotonic per project. Backed by .next-session-id.json; reconciled at ao start against sessions/ + git ls-remote. .next-session-id.json.
3 — Tmux name tmuxName Always = sessionId. No prefix, no hash, no doubling. One rule, no exceptions. Runtime only.
3b — Branch name branch session/{sessionId} (worker), orchestrator/{sessionId} (orchestrator), or tracker-supplied if branch-safe. git.
3c — Worktree path workspacePath projects/{projectId}/worktrees/{sessionId}. No ~/.worktrees/. filesystem.
4 — Observability obs path projects/{projectId}/observability/processes/{component}-{pid}.{json,ndjson}. Cross-cutting / unattributed → ~/.agent-orchestrator/observability/orphan/. filesystem.
5 — Hashes that stay HMAC, feedback dedupe, kimicode md5 Webhook signing (security), feedback dedupe key (content addressing), kimicode mirror of external scheme. Unchanged.
Why keep V2 {basename}_{hash10}

The V2 format is right; we just need to apply it uniformly

  • Already in use for 6 of 8 projects. The migration footprint is small: only the 2 V1 bare-basename entries (agent-orchestrator, harshitsinghbhandari-github-io) need re-keying.
  • Readable on disk. ls projects/ still tells you what's there at a glance. Debug output, error messages, and grep all stay productive.
  • Deterministic from (path, origin). Re-registering at the same path finds the same projectId via the existing path-equality check at global-config.ts:674. No new lookup machinery needed.
  • 40 bits of hash is fine for a single-user CLI. Birthday collision risk starts around ~1M projects. The "please file a bug" recovery path can stay — it just won't fire.
  • Already filesystem-safe. sanitizeBasename at global-config.ts:55-62 already lowercases, strips non-[a-z0-9_-], caps at 30 chars.

What's not changing: the projectId format itself. The wins in this section come from the other rules — single prefix allocator, tmuxName ≡ sessionId, observability inside the project dir, stored counter — all of which are independent of project ID shape and apply cleanly on top of V2.

Why a stored counter for sessionId

Cheap, correct, monotonic

  • O(1) allocation instead of the current O(n) directory + remote scan.
  • Monotonic across upstream branch deletion. Delete session/ao-131 upstream and the next session is still ao-132, not a recycled ao-131.
  • Reconciler protects against drift. On ao start, scan sessions/*.json and git ls-remote; if max(observed) ≥ counter, advance. The remote scan is now a backstop, not the primary source.
  • One file, one lock. Atomic increment under the project's own write lock; cross-project allocations don't contend.

Phase 4

Execute — atomic, marker-driven, resumable

2 Same-repo merge prompt For each merge candidate, ask: keep both? merge sessions? Default = merge.
3 Path renames renameSync(projects/{oldId}, projects/{newId}). Atomic on local FS.
4 Write identity.json For each project: { projectId, displayName, sessionPrefix, originUrl, path, realpath, repo, defaultBranch, registeredAt, schemaVersion: 3 }.

  • No ~/.worktrees/ dirs remain (or, all leaves accounted for in some projects/{id}/worktrees/).

Rollback

  • Reads latest migrations/v3-{ts}.json.
  • Reverses path renames in reverse order.
  • Restores config.yaml from snapshot.
  • Re-creates per-config observability dirs from the merged data (best-effort — observability is non-critical and can be lost).
  • Restores *.migrated directories if present.

Rollback is for the "I ran this and the dashboard broke" case. The forward migrator's verification phase should make rollback rare.

Edge cases handled

What this catches that migrate-storage didn't

  • Stuck orchestrators with legacy tmux names in metadata (the user's orchestrator.json from §7).
  • Doubled-prefix tmux names like ao-ao-orchestrator — Phase 7 renames live and Phase 6 fixes metadata.
  • Orphan worktrees at ~/.worktrees/ — Phase 8 adopts them.
  • Observability directory leak — Phase 9 attributes by reading the obs JSON's own projectId field.
  • Dual-registration of the same repo (the user's ao+ao2) — Phase 2 prompts; default merges.
  • storageKey-stripping race from PR #1466 — wrapped in the global config lock.
  • Mid-migration crash — markers persist progress; resume on re-run.
  • FS sync paths (Dropbox/iCloud) — pre-flight warns; rename atomicity not guaranteed; user is told.
What this command intentionally won't do

Out of scope (kept for later)

  • Multi-tenant / hosted scaling (10-char hash widening, salting). Not relevant for a single-user CLI tool today.
  • Cross-machine identity reconciliation. Synced ~/.agent-orchestrator/ is documented as unsupported.
  • Deleting the dead exports themselves — the migrator only emits a manifest. Deletion is a separate PR with its own review.
  • Renaming legacy git branches (session/old-prefix-N) on remote. Branches are on GitHub; we don't rewrite remote refs.

New file

  "source": "ao-project-add"

Why a separate file (instead of embedding everything in config.yaml):

  • Locality of writes. Project-level state changes (rename, branch update, prefix re-allocation) happen on a single file under the project's lock — no global config contention.
  • Survives registry corruption. If config.yaml goes bad, identity is recoverable from each project dir. Reconcile rebuilds the registry from identity.json files.
  • One source of truth per concept. config.yaml becomes a thin index (projectId → path); identity.json holds the rest.

Why these fields specifically:

  • schemaVersion: 3 — gates future migrations. Migrator refuses unknown versions; bumps require explicit code review.
  • path + realpath stored separately — handles symlinks and case-insensitive FS without re-resolving on every read.
  • registeredBy — debug breadcrumb when something looks wrong six months from now ("which version of ao wrote this?").
  • sessionPrefix stored, not derived — the canonical fix for the ao/ao2 bug. No more localConfig.sessionPrefix short-circuit; the prefix is decided at registration and never recomputed.
New file

~/.agent-orchestrator/projects/{projectId}/.next-session-id.json

{
  "schemaVersion": 3,
  "next": 133,
  "lastReconciledAt": "2026-05-07T16:00:00Z",
  "reconcileSource": "sessions+remote",
  "highestObserved": {
    "sessions": "ao-132",
    "remoteBranches": "ao-129",
    "tmuxSessions": "ao-132"
  }
}

Why a counter file (instead of scanning every spawn):

  • O(1) vs O(n). Spawning ao-200 currently scans 200 session JSONs and runs git ls-remote. With a counter, it's an atomic increment.
  • Monotonic across upstream branch deletion. If somebody deletes session/ao-131 on GitHub after the session is archived, the counter still reads 132. No reuse.
  • Reconciler protects drift. On ao start, scan sessions/ + git ls-remote; if max(observed) ≥ next, advance and update highestObserved. Counter file is the source of truth; scans are the safety net, not the primary read path.
  • Crash safety. Atomic write under the project's lock. If a crash leaves it stale, the next start reconciles. Worst case: counter advances by an extra step. Never reuses a number.

Why highestObserved: debugging. When something allocates an unexpected number, you can see which source disagreed with the counter and act surgically.

Why a single record file:

  • Audit trail. Six months from now, "why is this projectId different from what I expected?" has a written answer.
  • Rollback driver. ao migrate --rollback reads this file in reverse to undo the work. No guessing.
  • Postmortem source. If a migration goes wrong, the record + the snapshot config + the markers are enough to reconstruct any step.
  • Bounded retention. expiresAt drives a sweep — old records get archived, not kept forever.

Why deadExportsManifest: the migrator doesn't delete the dead exports itself (deletion is risky and deserves its own PR + review). It produces a list, ready to feed codemod or a manual deletion patch. The list is computed by grepping at migrate time — if a future commit makes one of these live again, it won't be on the list.

Defense points:

  • Resumability. Crash mid-step? Re-running ao migrate reads this file, sees currentStep=7, inProgressStep, retries from step 7.

Cross-cutting defense

Three defense points to lead the team review with

  • "This isn't a redesign — it's the missing other half of PR #1466." V2 introduced the storage layout but didn't finish the cleanup. storageKey stayed in the schema. generateConfigHash kept the observability dir leak alive. Three prefix allocators kept disagreeing. V3 finishes the V2 promise.
  • "Migration is forward-compatible by design." Every new file carries schemaVersion: 3. The loader refuses unknown versions. The next migration (V4, whenever it happens) inherits the same plan/progress/record/rollback machinery — we're not building a one-off.
  • "Defense in depth on a single laptop, optionality for hosted later." The whole proposal is single-user safe right now. The places that need to change for hosted multi-tenant — wider hashes (P10), salting (P11), distributed counter (P9), cross-machine lock (P12) — are isolated to specific files (generateExternalId, .next-session-id.json, config.yaml lock). We don't pay for them today; we don't preclude them either.

Verdict — and What to Build Next

AO's cryptographic primitives are correct: SHA-256, HMAC with timingSafeEqual, normalized inputs, atomic writes. The system built on top of them is not. Three project-identity formats, four prefix-collision schemes, three live tmux conventions, 153 leaked observability directories, a dozen dead public exports, and metadata that points at directories that don't exist. This isn't a hash bug. It's a layering bug — content-addressed identity was used where a counter or an opaque ID would have served.

Score today: 72/100 Crypto: 90. Identity system: 60. Combined: 72. Held back almost entirely by surface-level inconsistency, not crypto choice.
The proposal collapses 4 layers into 1 rule each V2 {basename}_{hash10} kept and applied uniformly. One canonical prefix allocator. Stored counter for sessionId. tmuxName ≡ sessionId. Observability in the project dir.
ao migrate is one pass Inventory → plan → pre-flight → 12 atomic steps with markers → verify → record. Idempotent. Resumable. Rollback-aware.
After V3: ~85/100 Hash sites become surgical; identity system becomes coherent. Hosted/multi-tenant later may want to widen the 10-char hash, but everything below the projectId layer is structural and finally clean.
